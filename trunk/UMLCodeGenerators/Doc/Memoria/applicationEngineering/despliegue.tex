Una vez creados los generadores de código, el siguiente paso es empaquetarlos y distribuirlos de forma que puedan ser usados de la forma más cómoda posible por diferentes desarrolladores para la creación de productos concretos pertenecientes a nuestra familia de productos.

La forma más fácil de distribuir nuestra infraestructura es crear un plugin,
que se integre con la plataforma Eclipse, que permita la generación de un proyecto \emph{Visual Studio 2010} que contenga dos tipos de proyectos, en función del generador de código ejecutado. Se podría haber pensado en integrar los generadores de código en \emph{Visual Studio 2010} o un editor comercial de UML, pero ninguna de dichas herramientas ofrecía tales facilidades. Por ejemplo, desde \emph{Visual Studio 2010} resulta imposible invocar las plantillas de generación de código creadas. \emph{Visual Studio 2010} ofrece un lenguaje de generación de código, pero sus funcionalidades están a años luz de las ofrecidas por otros lenguajes de generación de código, como EGL.

En el caso de ejecutar el generador de código correspondiente a la fase de \emph{Ingeniería del Dominio}, se crearía un proyecto que contendría los esqueletos de las clases parciales que conformarían la \emph{implementación de referencia}. Dichos esqueletos, como ya hemos comentado, han de completarse a mano.

En el caso de ejecutar el generador de código correspondiente a la fase de \emph{Ingeniería de Aplicaciones}, se crearía un proyecto que contendría las clases parciales necesarias para componer las clases parciales adecuadas de la implementación de referencia, de acuerdo a las características seleccionadas.

Para integrar los generadores de código en Eclipse, se ha hecho uso de su entorno \emph{Plug-in Development Environment} (PDE)~\footnote{http://projects.eclipse.org/projects/eclipse.pde}. Utilizando este entorno, añadimos una serie de elementos gráficos a Eclipse para permitir invocar las plantillas de generación de código desarrolladas.

El código asociado a dichos elementos gráficos realizaba varias tareas: (1) en primer lugar, se preprocesaba el modelo UML 2.0 para eliminar elementos innecesarios, como perfiles no utilizados, comúnmente introducidos por los editores comerciales. A continuación, se invocan las plantillas EGL desde código Java.

Una vez desarrollado el plug-in, se procedió a su empaquetado (siguiendo las directrices impuestas por Eclipse), y a la creación del correspondiente instalador para Eclipse, conocido como \emph{update site}. Por último, se creó una página web para darle visibilidad al proyecto.
